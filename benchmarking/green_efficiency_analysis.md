# Measurement Methodology
To evaluate the execution efficiency of our C programs, we employed the standard library function `clock()` to capture the CPU time before and after specific code blocks. By calculating the difference between the end and start clock ticks, and subsequently dividing by the `CLOCKS_PER_SEC` constant, we derived the execution time in seconds. To ensure the reliability of our metrics and minimize the influence of background operating system tasks, each benchmark was executed multiple times. We then analyzed these repeated runs to establish a dependable average. In the multi-phase program experiment, we strategically positioned timing hooks around three core functions—data construction, mathematical processing, and checksum reduction—to create a granular breakdown of where the processor allocated its computational resources.

# Observed Performance Differences
The recorded outputs prominently illustrate the severe performance penalties associated with suboptimal algorithm design. In our comparative analysis using an array of 50,000 integers, the execution time of the `count_even_naive` function was substantially longer than its optimized counterpart. The naive algorithm's nested loops yielded an O(N^2) complexity, forcing the CPU to perform millions of unnecessary checks. Conversely, the `count_even_single_pass` function completed the identical logical objective in a fraction of a second by traversing the dataset only once. The metrics explicitly show the single-pass implementation operating at speeds exponentially faster than the naive approach. Additionally, our phase-based instrumentation revealed that the intensive arithmetic operations within the "PROCESS" phase consumed nearly the entirety of the execution time, eclipsing the minimal time required for the "BUILD_DATA" and "REDUCE" phases.

# Relation Between Runtime and Energy Consumption
There is a direct and undeniable link between CPU execution time and overall energy consumption. Processors draw electrical power continuously to maintain logic gates, access memory, and compute instructions. When the CPU is bound to an inefficient O(N^2) algorithm, it is locked into a high-performance, high-draw state for an extended period. The naive algorithm squanders energy on redundant calculations. By refactoring the logic into a single-pass O(N) execution, the CPU completes the workload rapidly and immediately returns to an idle state, which consumes significantly fewer watts. Thus, writing efficient, fast-executing code is a fundamental requirement for green computing, as lowering the runtime inherently reduces the power footprint of the hardware.

# Limitations of the Experiment
While the empirical data is highly illustrative, this benchmarking approach carries certain structural limitations. The `clock()` function measures CPU time, which may not perfectly align with the actual wall-clock time experienced by the user, especially in multi-threaded environments or under heavy system load. Furthermore, this experiment ignores the effects of memory hierarchy and cache utilization; a program might be slow due to frequent cache misses rather than pure instruction volume. Another significant limitation is the lack of physical power measurement. We inferred energy consumption theoretically from the runtime differences, but without a hardware power meter, we cannot provide an exact quantification of the joules saved by the optimized algorithm.

# Practical Engineering Takeaway
The core takeaway from these benchmarking exercises is that algorithmic complexity dictates performance boundaries that hardware cannot overcome. An engineer must always prioritize choosing the correct data structures and algorithms (e.g., O(N) over O(N^2)) before attempting micro-optimizations. Furthermore, the instrumentation exercise underscores the value of surgical profiling. Instead of blindly refactoring an entire codebase, developers should utilize timing boundaries to isolate exact bottlenecks. By identifying that the "PROCESS" phase is the primary consumer of CPU cycles, engineering teams can focus their optimization efforts where they will have the most substantial impact on both speed and energy efficiency.
