# Measurement Methodology
In this experiment, we utilized the standard C library function `clock()` from `<time.h>` to measure the CPU time consumed by various algorithmic implementations. This function returns the number of clock ticks elapsed since the program was launched, which we then divided by the `CLOCKS_PER_SEC` macro to convert the result into seconds. We isolated the computational workload by starting the timer immediately before the algorithm's execution and stopping it immediately after the algorithm returned its result. To account for potential system noise and background processes, we executed each program a minimum of three times, recording the outcomes in our metrics files. Taking the average of these repeated runs provided a more stable baseline for our execution metrics. Furthermore, during the instrumentation phase, we strategically placed measurement boundaries around distinct programmatic phases—specifically data generation, processing, and reduction—to identify the exact segments consuming the most time.

# Observed Performance Differences
Our empirical data reveals profound performance disparities when comparing different algorithmic complexities. As documented in the `comparison_algorithms-metrics.md` file, the naive approach, which utilizes nested loops to count even numbers, demonstrated an exponentially higher execution time compared to the single-pass implementation. The nested iteration results in an O(N^2) time complexity, forcing the processor to perform redundant evaluations. In stark contrast, the single-pass algorithm iterates through the dataset exactly once. The timing metrics clearly show that the single-pass algorithm is orders of magnitude faster. Furthermore, the instrumentation lab showcased how breaking down a program into discrete phases allows us to pinpoint bottlenecks. The processing phase, involving multiple arithmetic operations, consistently demanded more CPU cycles than the relatively straightforward checksum reduction phase.

# Relation Between Runtime and Energy Consumption
The relationship between algorithmic runtime and energy consumption is fundamentally direct in the context of processor utilization. A CPU consumes electrical power to execute instructions and maintain memory state. When an algorithm like our naive nested-loop implementation runs significantly longer, it forces the CPU to remain in an active, high-power state for an extended duration. Conversely, the single-pass algorithm completes its task rapidly, allowing the CPU to return to an idle, low-power state. Green computing heavily relies on minimizing active CPU time. Therefore, reducing the execution time through algorithmic optimization directly translates to a reduction in joules consumed by the underlying hardware.

# Limitations of the Experiment
While our methodology provides a solid foundational understanding, it contains strict limitations. Primarily, the `clock()` function measures CPU time, not actual elapsed wall-clock time, and its resolution can vary across different architectures. Furthermore, our experiment does not isolate the impact of CPU cache hits and misses, which can drastically skew performance results. We also did not disable CPU frequency scaling, meaning the processor might have dynamically adjusted its clock speed during execution, introducing error into our timing consistency. Finally, measuring energy efficiency solely through execution runtime is an abstraction; we lacked hardware-level power profiling tools to measure actual wattage drawn.

# Practical Engineering Takeaway
The most critical engineering takeaway from this analysis is that hardware-level optimizations cannot rescue fundamentally flawed algorithmic designs. No amount of compiler tuning will close the performance gap between an O(N^2) and an O(N) algorithm at scale. Engineers must prioritize algorithmic complexity as the primary lever for performance and green efficiency. Additionally, targeted instrumentation is essential. By methodically wrapping distinct functional phases with precise timing boundaries, developers can accurately identify bottlenecks and focus optimization efforts on the specific code segments that yield the highest return on investment.
